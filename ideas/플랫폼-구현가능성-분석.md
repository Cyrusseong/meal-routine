### 1.2 한국 시장 트렌드

| 트렌드 | 상세 |
|--------|------|

## 3. 커머스 API 연동 가능성

### 3.1 쿠팡 파트너스

| 항목 | 상세 |
|------|------|
| **공식 API** | ✅ 존재 — developers.coupangcorp.com |
| **가입 조건** | 2단계: 계정 생성 → 첫 매출 발생 시 최종 승인 |
| **API 활성화 조건** | 판매물품 총 금액 **15만원 이상** 달성 후 API 키 발급 가능 |

| **MVP 우회 전략** | API 없이도 쿠팡 검색 URL 자동 생성으로 장보기 연동 가능 |
| **실현 가능성** | ⭐⭐⭐⭐ — MVP는 검색 링크로 시작, 매출 발생 후 API 전환 |

| **쿠팡이츠** | 쿠팡 파트너스에 포함 가능 | 쿠팡이츠 검색 URL 생성 | ⭐⭐⭐⭐ |

### 3.5 듀얼 커머스 연동 종합 전략

```
경로 A (직접 요리 → 식재료 어필리에이트):
  Phase 1 (MVP):    쿠팡/컬리/B마트 검색 URL 자동 생성 → 사용자가 직접 구매
  Phase 2 (성장기):  쿠팡 파트너스 API 연동 (어필리에이트 수익 확보)
  Phase 3 (확장기):  마켓컬리·B마트 사업 제휴 → 직접 장바구니 연동

경로 B (바로 주문 → 음식 주문 어필리에이트):
  Phase 1 (MVP):    배민/쿠팡이츠 검색 URL + 건강 메뉴 추천
  Phase 2 (성장기):  쿠팡이츠 파트너스 API 연동
  Phase 3 (확장기):  배민 사업 제휴 → 건강 식단 카테고리 협업
```

## 4. 레시피 파싱 기술 분석

### 4.1 한국 레시피 사이트 구조화 데이터 현황

| 사이트 | JSON-LD (Schema.org/Recipe) | HTML 구조 | 파싱 난이도 |
|--------|---------------------------|-----------|------------|
| 만개의레시피 | ❌ 미적용 | BeautifulSoup 크롤링 필요 | 중 |
| 해먹남녀 | ❌ 미적용 | HTML 파싱 필요 | 중 |
| 해외 레시피 사이트 | ✅ 대부분 적용 | JSON-LD 파싱 가능 | 하 |

**핵심 이슈:** 한국 사이트는 JSON-LD 구조화 데이터 채택률이 극히 낮음 (조사 대상 208개 국내 사이트 중 JSON-LD 사용 0건). 크롬 익스텐션에서 HTML 직접 파싱 또는 AI 기반 추출이 필요.

### 4.2 레시피 파싱 기술 스택 옵션

| 방식 | 장점 | 단점 | 추천도 |
|------|------|------|--------|
| **JSON-LD 파싱** | 표준화, 정확도 높음 | 한국 사이트 미지원 | 해외 사이트 전용 |
| **HTML 크롤링** | 모든 사이트 대응 | 사이트별 파서 개발 필요 | ⭐⭐⭐ |
| **AI 기반 추출** | 범용적, 구조 변경에 강함 | 비용 발생, 정확도 변동 | ⭐⭐⭐⭐ (장기) |
| **하이브리드** | JSON-LD 우선 → AI 폴백 | 구현 복잡도 증가 | ⭐⭐⭐⭐⭐ |

**추천:** 해외 사이트는 JSON-LD 파싱, 한국 사이트는 주요 5~10개 사이트용 HTML 파서 + AI 폴백의 하이브리드 전략.

## 6. 추천 플랫폼 전략

### 6.1 Phase별 로드맵

#### Phase 1: MVP 검증 (크롬 익스텐션 + PWA)

```
┌─────────────────────────────────────────────────────┐
│  크롬 익스텐션 (React + TypeScript + Vite)           │
│  ├── 웹 레시피 원클릭 저장                             │
│  ├── 재료 자동 추출 (HTML 파서 + AI 폴백)             │
│  └── Supabase에 레시피 데이터 저장                     │
│                                                       │
│  PWA 웹앱 (Next.js + Supabase)                       │
│  ├── 주간 식단 캘린더 (드래그 앤 드롭)                  │
│  ├── 장보기 리스트 (재료 자동 취합)                     │
│  ├── 쿠팡/컬리 검색 링크 자동 생성                      │
│  └── 반응형 모바일 최적화                               │
└─────────────────────────────────────────────────────┘
```

**이유:**
- 크롬 익스텐션은 "레시피 저장" 진입점으로 최적 (사용자가 레시피를 탐색하는 곳이 브라우저)
- PWA는 SEO 이점 + 앱 설치 없이 접근 가능 + 개발 비용 최소
- Supabase 무료 티어로 MVP 비용 최소화
- Next.js의 SSR/SSG로 레시피 콘텐츠 SEO 극대화

#### Phase 2: 성장기 (모바일 앱 확장)

```
┌─────────────────────────────────────────────────────┐
│  React Native 모바일 앱                               │
│  ├── PWA 기능 이식 + 네이티브 UX                       │
│  ├── 푸시 알림 (식사 리마인더)                          │
│  ├── 오프라인 장보기 리스트                              │
│  └── 쿠팡 파트너스 API 연동 (수익화)                    │
│                                                       │
│  기존 크롬 익스텐션 + PWA 유지                          │
└─────────────────────────────────────────────────────┘
```

**이유:**
- React Native는 JavaScript 인력풀이 넓고, PWA 코드의 로직 재사용 가능
- 앱스토어 존재감 확보로 사용자 획득 채널 확대
- 쿠팡 파트너스 매출 15만원 달성 후 API 전환

#### Phase 3: 확장기

```
┌─────────────────────────────────────────────────────┐
│  AI 기능 강화                                         │
│  ├── 냉장고 파먹기 모드 (pgvector 유사도 검색)          │
│  ├── 개인 맞춤 식단 추천                                │
│  └── 영양소 자동 분석                                   │
│                                                       │
│  커머스 통합                                           │
│  ├── 마켓컬리/B마트 사업 제휴                           │
│  └── 원클릭 장바구니 담기                               │
└─────────────────────────────────────────────────────┘
```

### 6.2 추천 기술 스택 총정리

## 7. 데이터 출처 및 저작권 구조

### 7.1 레시피 저작권 법적 분석

한국 저작권법상 레시피(요리법)의 보호 범위:

| 구분 | 저작권 보호 여부 | 근거 |
|------|-----------------|------|
| **레시피 자체 (재료·순서·방법)** | ❌ 보호 안 됨 | 아이디어/표현 이분법 — 레시피는 "아이디어"에 해당 |
| **레시피의 창작적 서술** | ✅ 보호 가능 | 독창적 문장 표현은 "어문저작물"로 인정 |
| **요리 사진·영상** | ✅ 보호됨 | 사진저작물·영상저작물 |
| **요리책의 편집·구성** | ✅ 보호됨 | 편집저작물 |

**실무적 함의:**
- 재료명·분량·조리 순서를 **자체 표현으로 재구성**하면 저작권 침해 아님
- 원본 사진·서술을 그대로 복제하면 침해 가능
- 크롬 익스텐션은 **사용자가 자발적으로 저장**하는 구조 → 서비스의 직접 크롤링이 아니므로 법적 리스크 최소

### 7.2 출처 표시가 가능한 공공 데이터 소스

한국 정부 공공데이터포털(data.go.kr) 및 식품안전나라에서 **무료·출처 명시 가능한** 데이터를 제공:

#### 레시피 데이터 (출처: 공공데이터포털)

| API 이름 | 제공기관 | 주요 내용 | URL |
|----------|---------|----------|-----|
| **조리식품의 레시피 DB** | 식품의약품안전처 | 조리식품 레시피 정보 | data.go.kr/data/15060073 |
| **레시피 기본정보** | 농림수산식품교육문화정보원 | 요리명, 조리시간, 난이도, 칼로리 | data.go.kr/data/15057205 |
| **레시피 재료정보** | 농림수산식품교육문화정보원 | 재료명, 분량, 영양성분, 원산지 | data.go.kr/data/15058981 |
| **레시피 조리과정** | 농림수산식품교육문화정보원 | 단계별 조리방법, 팁 | 공공데이터포털 |

#### 영양 데이터 (출처: 식품의약품안전처)

| API 이름 | 주요 내용 | URL |
|----------|----------|-----|
| **식품영양성분DB정보** | 식품명, 분류, 에너지, 탄수화물, 단백질, 지방 등 전체 영양소 | data.go.kr/data/15127578 |
| **전국통합식품영양성분 표준데이터** | 농수축산물+가공식품 통합 영양 정보 (26개 영양소 컬럼) | data.go.kr/data/15100064 |
| **식품영양성분 데이터베이스** | 식약처 DB — 맞춤 영양 진단 모델 포함, 소스코드 공개 | various.foodsafetykorea.go.kr/nutrient |

**활용 조건:** 공공데이터포털 회원가입 → API 활용 신청 → 인증키 발급 → 무료 사용 (출처 표기 필수: "출처: 공공데이터포털" 또는 해당 기관명)

### 7.3 레시피 출처 관리 아키텍처

```
┌──────────────────────────────────────────────────────────────────┐
│  레시피 데이터 소스                                                 │
│                                                                    │
│  Layer 1: 공공 데이터 (출처 명확)                                   │
│  ├── 식약처 조리식품 레시피 DB (공공데이터포털 API)                    │
│  ├── 농림수산식품교육문화정보원 레시피 API                             │
│  └── 출처: "공공데이터포털 / 식품의약품안전처" 자동 표기                 │
│                                                                    │
│  Layer 2: 사용자 저장 레시피 (크롬 익스텐션)                          │
│  ├── 사용자가 웹에서 원클릭 저장                                      │
│  ├── 원본 URL 자동 저장 → 출처 링크로 활용                           │
│  ├── 재료·순서를 구조화 데이터로 파싱 (자체 표현)                      │
│  └── 원본 사진은 저장하지 않음 → 원본 URL 링크만 보존                  │
│                                                                    │
│  Layer 3: 영양 정보 (출처 명확)                                     │
│  ├── 식약처 식품영양성분DB API → 재료별 영양소 자동 매칭               │
│  └── 출처: "식품의약품안전처 식품영양성분DB" 자동 표기                  │
│                                                                    │
│  Layer 4: 사용자 생성 레시피 (UGC)                                  │
│  └── 사용자가 직접 입력 → 출처 = "사용자 직접 등록"                    │
└──────────────────────────────────────────────────────────────────┘
```

### 7.4 DB 스키마 — 출처 관리 필드

```sql
-- recipes 테이블의 출처 관련 컬럼
CREATE TABLE recipes (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title TEXT NOT NULL,
    -- ...기본 필드...

    -- 출처 관리 필드
    source_type TEXT NOT NULL CHECK (source_type IN (
        'public_api',      -- 공공데이터 API
        'user_import',     -- 크롬 익스텐션으로 사용자 저장
        'user_created'     -- 사용자 직접 입력
    )),
    source_url TEXT,                -- 원본 URL (크롬 익스텐션 저장 시)
    source_name TEXT,               -- 출처 표시명 ("만개의레시피", "식약처" 등)
    source_api_id TEXT,             -- 공공 API의 원본 레코드 ID
    source_license TEXT,            -- 라이선스 정보 ("공공누리 제1유형" 등)

    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 영양 정보의 출처 관리
CREATE TABLE nutrition_info (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    recipe_id UUID REFERENCES recipes(id),
    -- ...영양소 필드...

    data_source TEXT DEFAULT '식품의약품안전처 식품영양성분DB',
    data_source_url TEXT DEFAULT 'https://various.foodsafetykorea.go.kr/nutrient/',
    last_synced_at TIMESTAMPTZ      -- 공공 데이터 동기화 시점
);
```

## 9. 결론

### 핵심 판단

1. **듀얼 어필리에이트 구조가 핵심** — 건강 목표 진입 → 직접 요리(식재료 어필리에이트) + 바로 주문(음식 어필리에이트) 양쪽 수익화
2. **시장 타이밍이 유리함** — 직접 요리 트렌드 + 배달 시장 동시 성장 + 통합 서비스 부재
3. **MVP 구현 난이도가 낮음** — 크롬 익스텐션 + PWA 조합으로 소규모 팀이 충분히 구현 가능
4. **크롬 익스텐션 + PWA가 최적** — 사용자가 레시피를 찾는 곳(브라우저)에서 시작하여, 별도 앱 설치 없이 식단 관리까지 연결
5. **커머스 연동은 단계적 접근** — 양쪽 모두 검색 URL 방식으로 시작, 트래픽 확보 후 API/사업 제휴로 전환
6. **Supabase + Next.js가 기술 스택 최적해** — 관계형 데이터 모델, SEO, 비용 효율성, AI 확장성 모두 충족
